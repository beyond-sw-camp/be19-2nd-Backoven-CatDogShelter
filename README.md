## 📢 프로젝트 소개
> 댕냥쉼터는 유기견·유기묘 입양과 봉사를 한곳에서 연결하는 커뮤니티형 플랫폼입니다.  
> 보호소 운영 주체(보호소장)와 시민 봉사자가 같은 공간에서 공지·모집·후기·신고·소통을 수행하도록 설계해,  
> 기관과 개인이 함께 운영하는 신뢰 기반 플랫폼을 지향합니다.

<br>

---

## ✨ 차별성
### 🙋🏻‍♀️ 봉사자 참여
- 봉사자 등급제(발자국 → 댕냥친구 → 댕냥지킴이 → 댕냥보호천사) 도입
- 매달 봉사왕 선정 (월별/누적 봉사시간 기반 매달 1일에 랭킹)
<img width="551" height="245" alt="image" src="https://github.com/user-attachments/assets/1babf266-f190-4cf0-81de-a2b7ae972f11" />

### 🤝🏻 보호소 운영 신뢰성 강화
- 사업자·법인 정보, 개업/폐업일자 입력을 통해 공신력 있는 보호소 운영 주체를 구분합니다.
- 단순 사용자 커뮤니티를 넘어 기관-개인 협업 구조를 구현합니다. 

### 📋 기부 및 후원 투명성
- 보호소장이 직접 후원계좌/기부내역을 관리·공개해 신뢰를 강화합니다.

### 🐕 개인 간 분양  
- 보호소에서 올린 입양공고뿐만 아니라, 개인과 개인 간 분양도 가능

| 변민철  | DB프로젝트 이후 본격적인 애플리케이션을 개발하는 경험이었다. 사실 Java문법을 배울 때 까지만해도 객체지향언어가 무엇인지에 대해 크게 와닿지 않았던 것 같다. 특히, 의존성과 같은 용어가 소프트웨어 개발하는 데에 있어서 어떤 의미인지도 감이 오지 않았다. 하지만 스프링을 배우고 나서 점차 '의존성을 주입받는다.', '다른 클래스를 의존한다' 등의 객체지향프로그래밍에서의 다양한 상황을 마주하면서 그 개념을 조금은 깨달은 것 같기도 하다. 이번 프로젝트를 하면서 가장 크게 배운 점은 소위 '뇌 뺴고' 개발을 진행한다면 미래에 마주할 에러에 크게 데일 수 있다는 것을 느꼈다. 왜 소프트웨어 개발하는 과정을 건축물에 비유하셨는지가 절실히 느껴졌다. 단순히 IDE 툴을 가지고 자바 코드를 입력하는 행위 자체가 프로그래밍인 줄 알았던 사람이었다. 하지만 단순히 그런 기술적인 작업/업무만이 아닌, 협업을 하면서 팀 내부의 업무/작업의 원칙과 기준을 수립하고  그것을 어떤 방식으로 지켜나갈지를 잘 정하는 것이 중요하다고 느꼈다. 가령 단순하게만 생각했던 Class이름을 서로 협의 없이 임의로 지었다가 Spring Duplicate Bean Definition 문제를 마주했고 그로 인해 다른 패키지나 클래스에 있는 코드를 수정해야했던 경험이 있었다. 이렇게 아주 사소한 것부터 팀 프로젝트 작은 규칙부터 기준까지, 체계를 세우고 프로젝트를 진행하는 것이 가장 중요하다고 생각했다. 물론 기술적인 부분에서 더욱 성장하는 것이 중요했지만, 이번 프로젝트에서 느낀 것은 잘 갖추어진 체계 속에서 원칙을 지키고 개발을 진행하다보면 기술적인 역량은 개발 중 기술적인 필요성에 의해 자연스럽게 따라올 것이라고 생각했다. Mybatis는 Mapper.xml 파일 내 mybatis내 문법을 숙지하고 동적쿼리를 잘 이해하는, 어떻게 보면 sql의 연장선이라고 이해했다. 그리고 Jpa는 시간이 너무 부족한 나머지 추가공부가 필요하다는 것을 절실히 느꼈다. 가볍게 이해한 것은 Entity클래스와 테이블간의 관계, 그리고 persistence context에 대해 조금 더 공부할 필요성을 인지했다. 그리고 JpaRepository의 기능(?) 혹은 개념과 특징을 이해하는 것이 앞으로의 과제를 수행하는 것에 큰 도움이 될 것이라는 성찰을 얻었다. 결론, 기술적인 내용이 아닌 소프트웨어 개발 전 프로세스에 대한 충분한 개념과 이해가 필요하다는 것이 이번 프로젝트의 가장 큰 깨달음이 아닐까 싶다. |
| 이승건  | (회고 내용) |
| 이지윤  | 이번 프로젝트에서는 자유게시판과 보호소장 파트를 담당하였다. 자유게시판의 경우, MyBatis를 사용하여 조회 기능을 구현하고, JPA를 활용해 데이터 삽입·수정·삭제를 처리하였다. 조회에는 MyBatis를 택한 이유가, 직접 SQL 쿼리를 작성함으로써 원하는 조건과 조인을 세밀하게 표현할 수 있었기 때문이다. 반면 JPA는 엔티티 기반으로 매핑해두면 기본적인 CRUD 쿼리를 자동 생성해주어 생산성이 높아, 삽입·수정·삭제에 적합했다. 다만, 두 기술을 혼합하다 보니 SQL 직접 작성 시 파라미터 매핑 오류가 발생하거나, 엔티티 설계 변경 시 JPA 쿼리가 의도와 다르게 생성되는 문제가 있었다. 이를 해결하기 위해, MyBatis는 Mapper XML에서 파라미터 이름과 DTO 필드명을 일치시키도록 규칙을 정했고, JPA는 @Query와 로깅을 병행하여 실제 실행되는 SQL을 확인하며 디버깅하였다. 보호소장 파트에서는 로그인 요청과 응답에 다른 DTO를 사용하여, 최초 로그인 시 JWT 토큰을 발급하고 이후 요청은 토큰 기반 인증으로 처리하도록 설계하였다. 또한 Spring Security를 적용해 회원 권한에 따라 접근 가능한 기능을 제한하였다. MSA 구조를 도입하여 모듈별 유지보수성을 높였으며, 일반 회원과 보호소장 회원의 Bean 충돌 문제는 클래스명에 접두사/접미사를 부여해 빈 이름을 구분함으로써 해결했다. 기능 테스트 중에는 권한 문제(HTTP 403)가 잦았는데, 이때는 Config 설정에서 permitAll()을 임시로 열어 기능만 검증한 뒤, 최종 병합 시 권한 정책을 다시 적용하는 방식을 사용했다. 또 GitHub 협업 과정에서 PR 병합 충돌이 빈번히 발생했는데, 이를 줄이기 위해 프로젝트 구조를 미리 문서화하고, 기능 개발 후 즉시 PR을 올리는 규칙을 팀 내에서 합의하여 문제를 해결하였다. |
