## 📢 프로젝트 배경 & 문제의식

현재 운영 중인 유사 서비스들은 대부분 **입양 정보 제공**에 초점을 맞추고 있으며,  
보호소 중심의 일방적인 정보 전달에 머무르는 경우가 많습니다.

이로 인해 **봉사자 참여를 체계적으로 지원하거나,  
기부 및 신고 기능을 제공하는 데 한계**가 존재합니다.

그러나 유기동물 보호와 입양 문화 확산을 위해서는 단순한 입양 정보 제공을 넘어,  
**봉사 활동 · 후원 관리 · 신고 접수 · 커뮤니티 소통**까지 함께 지원할 수 있는  
**통합 플랫폼의 필요성**이 대두되고 있습니다.

<br>

## 📝 서비스 개요
<img width="737" height="411" alt="Image" src="https://github.com/user-attachments/assets/c3bf2b4d-8520-4acf-aa39-6cd991759f04" />

> 댕냥쉼터는 **유기견·유기묘 입양과 봉사 활동을 한곳에서 연결**하는 커뮤니티형 플랫폼입니다. <br>
> **보호소 운영 주체(보호소장)와 시민 봉사자가 같은 공간에서 공지·모집·후기·신고·소통을 수행**하도록 구현해, <br>
> **기관과 개인이 함께 운영하는 신뢰 기반 서비스를 지향합니다.**

<br>

## ✨ 차별성
### 🙋🏻‍♀️ 봉사자 참여
- 봉사자 등급제(발자국 → 댕냥친구 → 댕냥지킴이 → 댕냥보호천사) 도입
- 매달 봉사왕 선정 (월별/누적 봉사시간 기반 매달 1일에 랭킹)
  <img width="600" height="300" alt="Image" src="https://github.com/user-attachments/assets/32bcc310-105b-429d-a178-7a51a34cf0d3" />

### 🤝🏻 보호소 운영 신뢰성 강화
- 사업자·법인 정보, 개업/폐업일자 입력을 통해 공신력 있는 보호소 운영 주체를 구분합니다.
- 단순 사용자 커뮤니티를 넘어 기관-개인 협업 구조를 구현합니다.

### 📋 기부 및 후원 투명성
- 보호소장이 직접 후원계좌/기부내역을 관리·공개해 신뢰를 강화합니다.

### 🐕 개인 간 분양
- 보호소에서 올린 입양공고뿐만 아니라, 개인과 개인 간 분양도 가능

## 📝 주요 기능

### 👤 회원 관리
- 일반 회원 / 보호소장 회원 구분
- 보호소장 회원: 사업자등록번호, 법인등록번호, 개업/폐업일자 입력 가능 (공신력 확보)

### 🏠 입양 게시판
- 보호소 및 개인 등록 입양 공고 조회
- 게시글 등록 / 목록 조회 / 상세 조회 / 이미지 첨부
- 최신순, 조회순, 추천순 정렬 지원
- 조건 검색 (동물종류, 지역, 상태 등)
- 댓글, 추천, 신고, 파일 업로드 기능

### 🔍 실종·목격 게시판
- 실종 신고 등록 및 키워드 검색
- 목격 제보 등록
- 댓글 작성 / 삭제 / 신고 기능

### 🤝 봉사 모임
- 보호소장이 직접 봉사 일정 등록 및 모집
- 봉사자 신청 / 승인 / 취소
- 후기 작성, 댓글 기능
- 봉사시간 자동 집계 및 **봉사 등급제**  
  (발자국 → 댕냥친구 → 댕냥지킴이 → 댕냥보호천사)
- 월별·누적 봉사왕 시스템

### 🎁 기부·후원
- 보호소장이 직접 후원 계좌, 기부 내역 등록/관리
- 기부 내역 공개를 통한 투명성 강화

### 📢 공지사항 게시판
- 보호소장 전용 공지 게시판
- 일반 회원 열람 가능

### 💬 커뮤니티
- 자유게시판, 후기 공유, 공지사항 운영
- 댓글, 추천, 신고, 파일 업로드 공통 지원

### 🚨 신고 시스템
- 게시글·댓글 신고 접수 및 처리
- 중복 신고 불가, 블라인드 누적 관리
- 관리자 모드에서 신고 내역 조회/처리

## 🏗️ MSA 아키텍처
- 댕냥쉼터는 MSA(Microservice Architecture) 기반으로 설계되어, 서비스별 독립성과 확장성을 보장합니다.
- 각 마이크로서비스는 Eureka Server에 등록되고, Spring Cloud Gateway를 통해 라우팅 및 보안 처리를 수행합니다.
  <img width="1740" height="781" alt="image" src="https://github.com/user-attachments/assets/9ec427cc-e083-4888-abc1-95abe91a4053" />

#### CATDOGSHELTER-USER-SERVICE
- 회원 관리 및 보호소장 관리 기능 담당
- Spring Security와 연동, 로그인/회원가입/권한 관리 수행

#### CATDOGSHELTER-POST-SERVICE
- 입양 게시판, 봉사 모집글, 신고 등 게시글 관련 기능 처리
#### GATEWAY-SERVER
- 사용자 요청을 받아 각 서비스로 라우팅
- AuthorizationHeaderFilter를 적용하여 JWT 기반 인증/인가 처리

## 📃 프로젝트 회고록
| 조원명 | 회고 |
|--------|-------|
| 곽동근  | 이번 프로젝트는 Spring Boot 기반 MVC를 기본으로 하고, DDD설계를 채용하고 CQRS패턴을 적용하여 백엔드 프로젝트를 진행하였는데 프로젝트 주제에 대한 도메인이 여러개 나눠져있어서 열심히 배웠던 내용들을 각자 최대한 활용해서 팀원들이 각자 맡은 모듈에 잘 녹일 수 있었던 것 같다. 그리고 다들 열정적이고 능동적으로 잘 움직이는 사람들이 모여있어서 프로젝트 진행하는동안 일정에 대한 차질도 없었고 서로 모르는 부분들은 보완해가면서 성장하고 프로젝트가 잘 마무리 될 수 있었다. 그리고 프로젝트 거의 끝나갈무렵 security, MSA 아키텍처를 학습 한 후에 설계해서 프로젝트를 확장했는데 어렵고 이해가 안가는 부분들도 있어서 이 부분은 따로 학습을 해보고, 이 부분 외로 시간을 내서 백엔드 공부를 좀 더 해봐야될것같다.그리고 이번 프로젝트 일정에 대한 차질이 없긴했지만 아무래도 빠듯한 일정이였다보니 신경을 못 쓴 부분들에 대해서 트러블슈팅이 발생했었다.트러블슈팅을 겪은 첫번째로는, 처음 패키지 구조 설계할때 구조에 대한 설계뿐만 아니라  각 도메인별로 클래스까지 생성해둔 상태로 패키지 구조를 뿌리고 각자 개발했었는데 각 도메인별로 스프링 컨테이너에 Bean 이름이 등록될때 이름이 겹치면 안되는데 Entity, Controller, repository, Service 클래스 이름들을 전부 같게 해뒀었어서 나중에 모듈을 합쳤을때 쉽지않았었다. 그래도 초반에 회의를 진행하고 각자 맡은 도메인 클래스들 이름을 빠르게 수정해서 바로잡았고 다음으로 겪은문제는 첫번째때 겪은 문제를 나름 빨리 잡았다 생각했었는데 첫번째에 겪은 문제+모두 중간 중간 개발하고 PR날렸어야됐는데 이런 과정들이 많이 없었어서 병합 충돌이 크게 일어났었고 해당 부분들을 수정하는데 시간이 좀 오래 걸리긴했지만 좋은 경험과 큰 깨달음을 얻은 프로젝트 였던것같다. |
| 변민철  | DB프로젝트 이후 본격적인 애플리케이션을 개발하는 경험이었다. 사실 Java문법을 배울 때 까지만해도 객체지향언어가 무엇인지에 대해 크게 와닿지 않았던 것 같다. 특히, 의존성과 같은 용어가 소프트웨어 개발하는 데에 있어서 어떤 의미인지도 감이 오지 않았다. 하지만 스프링을 배우고 나서 점차 '의존성을 주입받는다.', '다른 클래스를 의존한다' 등의 객체지향프로그래밍에서의 다양한 상황을 마주하면서 그 개념을 조금은 깨달은 것 같기도 하다. 이번 프로젝트를 하면서 가장 크게 배운 점은 소위 '뇌 빼고' 개발을 진행한다면 미래에 마주할 에러에 크게 데일 수 있다는 것을 느꼈다. 강사님께서 왜 소프트웨어 개발하는 과정을 건축물에 비유하셨는지가 절실히 느껴졌다. 단순히 IDE 툴을 가지고 자바 코드를 입력하는 행위 자체가 프로그래밍인 줄 알았던 사람이었다. 하지만 단순히 그런 기술적인 작업/업무만이 아닌, 마치 건설을 할 때 설계를 잘하고 하고 토대를 잘 다져놓는 작업을 하는 것처럼 협업을 하면서 팀 내부의 업무/작업의 원칙과 기준을 수립하고 그것을 어떤 방식으로 지켜나갈지를 잘 정하는 것이 중요하다고 느꼈다. 가령 단순하게만 생각했던 Class이름을 서로 협의 없이 임의로 지었다가 Spring Duplicate Bean Definition 문제를 마주했고 그로 인해 다른 패키지나 클래스에 있는 코드를 수정해야했던 경험이 있었다. 이렇게 아주 사소한 것부터 팀 프로젝트 작은 규칙부터 기준까지, 체계를 세우고 프로젝트를 진행하는 것이 가장 중요하다고 생각했다. 물론 기술적인 부분에서 더욱 성장하는 것이 중요했지만, 이번 프로젝트에서 느낀 것은 잘 갖추어진 체계 속에서 원칙을 지키고 개발을 진행하다보면 기술적인 역량은 개발 중 기술적인 필요성에 의해 자연스럽게 따라올 것이라고 생각했다. Mybatis는 Mapper.xml 파일 내 mybatis내 문법을 숙지하고 동적쿼리를 잘 이해하는, 어떻게 보면 sql의 연장선이라고 이해했다. 그리고 Jpa는 시간이 너무 부족한 나머지 추가공부가 필요하다는 것을 절실히 느꼈다. 가볍게 이해한 것은 Entity클래스와 테이블간의 관계, 그리고 persistence context에 대해 조금 더 공부할 필요성을 인지했다. 그리고 JpaRepository의 기능(?) 혹은 개념과 특징을 이해하는 것이 앞으로의 과제를 수행하는 것에 큰 도움이 될 것이라는 성찰을 얻었다. 결론, 기술적인 내용이 아닌 소프트웨어 개발 전 프로세스에 대한 충분한 개념과 이해가 필요하다는 것이 이번 프로젝트의 가장 큰 깨달음이 아닐까 싶다. |
| 이승건  | 이번 프로젝트는 유기동물 커뮤니티 백엔드 개발을 목표로 진행하였다. 아키텍처는 CQRS 패턴을 적용하여 Query 파트는 MyBatis로, Command 파트는 JPA로 구현하였다. REST API를 지향하며 메소드의 기능에 맞게 매핑을 진행했으나, URL 명칭까지 완전히 RESTful하게 적용하지 못한 점은 아쉬움으로 남는다. 프로젝트 초기에 DDD 설계를 진행한 덕분에 개발 과정에서 흐름을 놓치지 않고 팀원들과 원활하게 소통할 수 있었다. 도메인 언어를 통일함으로써 개발 방향성을 공유하는 것이 얼마나 중요한지 체감할 수 있었다. DDD 기반으로 여러 종류의 게시판이 도출되었고, 팀원 모두가 DB-Java CRUD 경험을 쌓을 수 있도록 각자 하나 이상의 게시판을 맡았다. 나는 목격 정보 게시판, 등급·봉사왕 기능, 쪽지 기능을 담당하였다. 기능 분배 당시에는 게시판 외의 기능을 맡고 싶어 자발적으로 선택했지만, 상대적으로 가벼운 기능이어서 다른 게시판을 맡아볼 걸 하는 아쉬움도 있었다. 협업 과정에서는 규칙 부재로 인해 클래스명이나 파일명 충돌이 빈번하게 발생했다. 이를 해결하기 위해 도메인명을 접두사로 붙이는 방식을 사용했지만, 애초에 규칙을 정하지 않은 것이 문제의 원인이었다. 이 경험을 통해 DDD와 같은 설계 방식이 팀의 언어를 통일하고 방향성을 잡는 데 중요하다는 점, 그리고 협업을 위한 명확한 규칙 설정이 필수적이라는 점을 배울 수 있었다. |
